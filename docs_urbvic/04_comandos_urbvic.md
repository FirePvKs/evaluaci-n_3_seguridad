# 04. Análisis de Vulnerabilidad: Inyección de Comandos (Command Injection)

## 1. Evidencia del Ataque
* **Formulario Afectado:** Portal de Clientes - Sección "Command Injection".
* **Payload Utilizado:** `127.0.0.1; cat /etc/passwd`.
* **Resultado Obtenido:** El sistema ejecutó el comando de red original (`ping`) y, debido a la falta de validación del carácter delimitador `;`, procedió a ejecutar el comando inyectado `cat /etc/passwd`, exponiendo en pantalla el archivo de configuración con el listado de usuarios del sistema operativo del servidor.

## 2. Explicación Técnico-Operativa (¿Por qué funciona?)
La vulnerabilidad se produce porque la aplicación web de CrediExpress toma una entrada proporcionada por el usuario (en este caso, una dirección IP para validar conectividad) y la pasa directamente a una función del sistema operativo (como `exec()`, `system()` o `shell_exec()`) a través de un shell del sistema, sin sanitizar los metacaracteres de control[cite: 1, 3].

En el backend, el código ejecuta una instrucción similar a esta de forma insegura:
```bash
ping -c 4 $USER_INPUT
Al ingresar el payload 127.0.0.1; cat /etc/passwd, el intérprete de comandos de Linux procesa el punto y coma (;) como un separador secuencial de instrucciones. Por lo tanto, el servidor ejecuta primero el comando legítimo (ping -c 4 127.0.0.1) y, justo después, ejecuta el comando arbitrario del atacante (cat /etc/passwd) bajo los mismos privilegios del proceso del servidor web.

4. Política de Prevención (Causa Raíz)
Para erradicar por completo la inyección de comandos en el entorno operativo de CrediExpress, se deben aplicar las siguientes políticas de desarrollo seguro:

Evitar la Ejecución Directa de Comandos de Shell: La mejor defensa es no invocar funciones que llamen directamente al shell del sistema operativo. Si se requiere realizar una validación de red (como un ping), se deben utilizar APIs nativas del lenguaje de programación o librerías específicas de red en lugar de ejecutar binarios del sistema operativo.

Uso de Funciones de Ejecución Segura (Sin Shell): Si la invocación es estrictamente necesaria, se deben usar funciones que pasen los argumentos como un arreglo de cadenas aisladas en lugar de una única cadena de texto (por ejemplo, child_process.execFile o spawn en Node.js, configurando { shell: false }). Esto evita que el sistema interprete metacaracteres como ;, && o |.

Control de Mitigación (Defensa en Capas)Como medidas defensivas y salvaguardas perimetrales complementarias basadas en los marcos OWASP y CIS Controls, se establecen las siguientes directrices:  Principio de Menor Privilegio a Nivel de OS: Asegurar que el proceso del servidor web (por ejemplo, www-data o apache) se ejecute bajo una cuenta de usuario con privilegios extremadamente limitados dentro del sistema operativo. Esta cuenta jamás debe tener privilegios de superusuario (root) ni permisos de ejecución en directorios donde no corresponda.Hacking-Proofing mediante Listas Blancas (Input Validation): Implementar validaciones estrictas basadas en expresiones regulares en el backend antes de procesar cualquier dato. Si el campo espera una dirección IP, el sistema debe rechazar inmediatamente cualquier entrada que contenga caracteres que no sean números y puntos (ej. rechazando automáticamente ;, |, &, $, etc.).Aislamiento del Entorno (Containerización y Chroot): Ejecutar la aplicación web de la financiera dentro de un entorno aislado (como un contenedor Docker configurado como Read-Only en su sistema de archivos) o un entorno chroot. Esto limita drásticamente el daño potencial, impidiendo que el atacante acceda al sistema operativo anfitrión (Host) si logra comprometer el contenedor.
